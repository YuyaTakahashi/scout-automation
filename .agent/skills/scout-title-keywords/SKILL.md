# スカウトタイトル・キーワード集

オープンロジのPdM採用において、候補者の関心を惹きつけ、かつ事業内容や求める人物像と合致させるためのキーワード集です。

## 1. 開発手法・スタンス（プロセスの具体性）
「単に開発した」ではなく、PdMとしてどう動いたかを想起させるワードです。
- **アジャイル・スクラム**: 「スクラムで開発を創る」「アジャイルでUXを追求」
- **ユーザーインタビュー・仮説検証**: 「N=1から真の課題を抽出」「仮説検証でPMFを追求」
- **データ分析（SQL/BI）**: 「SQLで意思決定を支える」「データで顧客行動を変える」
- **Discovery（課題発見）**: 「ユーザーディスカバリー」「ユーザーリサーチ」

## 2. ビジネス・ドメイン特性（難易度の高さ）
オープンロジ特有の「物理が絡む複雑なB2B」に紐づくワードです。
- **B2B / エンタープライズ**: 「B2Bの深い負を紐解く」「複雑な業務フローを標準化」
- **サプライチェーン / SCM**: 「物流基盤をデジタルで創る」「ECと物流を滑らかに繋ぐ」
- **マルチステークホルダー**: 「三方良しのプラットフォーム設計」「多様な利害関係を調整し推進」
- **在庫・配送最適化**: 「実在庫と情報の同期を追求」「配送効率を最大化する設計」
- **DX / 新規事業**: 「物流のDXを牽引」「新規事業の起点を創る」

## 3. フェーズ・役割（0→1 / 10→100）
その人がどの「山」を登ってきたかを示すワードです。
- **新規立ち上げ / 0→1**: 「新規事業の0→1を完遂」「プロダクトの起点を創る」
- **グロース・基盤刷新**: 「負債を解消し基盤を刷新」「10→100の成長を支える設計」
- **オーナーシップ**: 「プロダクトの未来を一任」「事業成長のボトルネックを解消」

## ガイドライン：スカウトタイトルの型
抽象的な表現を避け、**「具体的手段(How) + 成果(Outcome)」**を組み合わせるのが基本形です。また、候補者の**「やりたいこと(Aspirations)」を最優先**で反映させてください。

### 志向性によるキーワード変換ルール
- 「世の中を面白く、豊かにしたい」「社会の役に立ちたい」等、広範な志向性を持つ場合
    - キーワード：**【社会課題】**（例：【社会課題をDXで解決】）

### 良い例 (Methodology + Outcome)
- 【UXでPMFを追求】
- 【データ分析で顧客を動かす】
- 【スクラムで0→1を創る】
- 【UX設計で事業変革】
- 【B2Bの深部課題を紐解く】
- 【新規立ち上げを牽引】
- 【社会課題をDXで解決】

### 悪い例 (抽象的 / NG)
- 【事業を創る】（具体性がなく刺さらない）
- 【PMFを追求】（手段が不明）
- 【Webシステム開発】（抽象的すぎて介在価値が見えない）
- 【WEBサービス改善】（同様に抽象的）
- 【SaaSを改善】（SaaSは禁止ワード。より具体的な業務に触れること）
- 【システム開発を担当】（PdMとしての介在価値が見えない）
- 【物流を革新】（言葉が強すぎて実態が見えない）

## フォールバック規則
候補者のプロフィールがいずれのキーワードや「具体的手法」に明確に合致しない、あるいは訴求が難しい場合は、**デフォルトタイトルとして以下を使用**してください。

- **【社会課題×累計65億調達】**

## 禁止ワード（絶対遵守）
「WEBシステム」「Webシステム」「WEBサービス」「Webサービス」「人材」「求人」「SaaS」「グロース」「牽引」「推進」「改善」「投資」「会計」「広告」「海外事業」「海外」「グローバル」「Overseas」「Global」
